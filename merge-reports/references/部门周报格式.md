# 部门周报格式

## 总体要求
1. 部门周报用于汇总多份个人周报，输出面向部门管理视角的周报内容。
2. 部门周报不保留个人填写痕迹，不写具体完成人姓名，不写单项工作的开始时间和结束时间。
3. 部门周报按固定板块编排，板块顺序与个人周报一致：
   金融研发、寄递研发、管理赋能、数据分析、信息资源管理、队伍建设及业绩融合、下周计划。
4. 部门周报以“归纳、压缩、概括”为原则，不逐条照抄个人周报原文。

## 文件头要求

部门周报开头只写填报周期：

```md
# 部门工作周报

填报周期：2026-03-20 至 2026-03-26
```

说明：
- 使用汇总后的统一周期。
- 不写个人姓名，不写参与人员名单。

## 正文编写规则

1. 每个板块使用二级标题。
2. 每个板块按项目归并后输出，不按人员拆分，不按个人周报顺序罗列。
3. 同一项目由多人填写时，必须合并为 1 条摘要，不按人员拆分，不重复输出同一项目。
4. 归并后只保留两类核心信息：
   - 本周工作内容总结
   - 本周工作成果总结
5. `下周计划` 板块只保留下周计划安排，不写本周成果。

## 归并原则

### 1. 按板块归并
- 先按个人周报中的板块进行汇总，不跨板块合并。

### 2. 按项目归并
- 每个板块内按项目名称归并。
- 同一项目的多人内容必须合并成一条摘要，不重复列出相似动作。

### 3. 压缩表达
- 将相近动作合并表达，例如“开发接口、联调接口、修复接口报错”可归纳为“推进接口开发和联调，完成问题修复”。
- 将多个细碎结果合并为阶段性成果表达，例如“完成字段梳理、完成方案设计、完成首版开发”可归纳为“完成需求梳理、方案设计和首版功能实现”。
- 对仍在推进、尚未结束的事项，统一使用部门口径表达，例如“持续推进”“完成阶段性处理”“形成阶段性成果”，避免混用个人过程描述。

### 4. 去除冗余信息
- 不写完成人员姓名。
- 不写每条工作的起止时间。
- 不重复出现语义相同的工作内容和成果。

### 5. 保持信息边界
- 个人周报中属于本周完成的事项，写入对应板块的本周总结。
- 个人周报中尚未完成但仍在推进的事项，可以在对应项目总结中体现“持续推进”或“阶段性完成”。
- 个人周报中的 `下周计划` 只进入部门周报的 `下周计划` 板块，不混入本周总结。

## 输出格式

推荐使用以下结构：

```md
# 部门工作周报

填报周期：2026-03-20 至 2026-03-26

## 金融研发

1. 金融员工履职系统：围绕履职规则优化和定位偏差处理开展功能调整，完成相关逻辑修复、字段梳理和接口方案设计，部分内容已提交测试。

## 寄递研发

1. 车牌寄递辅助系统：持续推进订单接口优化及云邮便民查询服务对接，完成部分接口开发、联调问题排查和日志优化。

## 管理赋能

1. 文创产品进销存系统：完成销售小票打印功能开发及模板调整，并提交测试验证。

## 数据分析

1. 抖音指数分析：完成省寄递业务部抖音指数数据核实，输出异常数据核查结果。

## 信息资源管理

1. 系统运维：开展省内云平台主机巡检、补丁升级和底层加固，完成部分节点处理并形成巡检记录。
2. 数据提取：完成第三方对账、时限对标等相关底表和明细数据提取，并提供业务使用。

## 队伍建设及业绩融合

1. 需求研讨：围绕无人包裹柜等项目开展需求和实施方案研讨，形成接口分工及实施排期意见。
2. 材料编写：完成软件著作权申报材料和阶段汇报材料整理。
3. 培训开展：完成相关系统培训讲解和现场答疑支撑。

## 下周计划

1. 金融员工履职系统：继续完善履职提醒规则配置并跟进测试问题修复。
2. 车牌寄递辅助系统：继续推进云邮便民查询服务对接接口开发和联调。
3. 系统运维：继续完成剩余主机补丁升级、巡检复核和迁移前环境检查。
```

## 语言要求

1. 使用部门视角表述，避免明显的个人执行口吻。
2. 语言尽量凝练，避免逐行复述个人周报。
3. 每个项目优先写“做了什么、产出了什么、当前推进到什么程度”。
4. 对于没有内容的板块，直接写 `无`。
5. 每个板块内使用顺序列表输出，采用 `1. 2. 3.` 的编号格式，不使用无序列表。
