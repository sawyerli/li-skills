# validate-report

校验标准周目录中 `raw/` 的个人工作周报文件，检查格式、结构和内容是否符合规范，并输出问题清单。校验通过后，将合规文件从 `raw/` 移动到同级 `apv/`，供后续 `merge-reports` 直接汇总使用。对于未出现的业务板块，统一按 `无` 处理。

## 格式规范来源

校验依据：当前 skill 目录下的 `references/个人周报格式.md`。执行前必须先读取该文件，以其中的规则为准。

## 触发方式

用户提供待校验的标准周目录或其 `raw/` 目录，例如：

```
/validate-report month202603/week20260326/
/validate-report month202603/week20260326/raw/
```

目录使用约定如下：

- `raw/` 存放员工初始提交的原始周报文件
- `apv/` 存放校验通过的周报文件
- `dept/` 预留给 `merge-reports` 存放生成好的部门周报

执行时优先采用以下工作流：

1. 员工先将原始个人周报放入对应周目录的 `raw/`
2. 运行 `validate-report`
3. 校验通过的文件从 `raw/` 移动到 `apv/`
4. 校验不通过或文件名不符合要求的文件继续保留在 `raw/`

校验通过后的输出目录规则：

```text
month202603/
  week20260326/
    raw/
    apv/
    dept/
```

其中：

- `202603` 取自 `填报周期结束` 的年月
- `20260326` 取自 `填报周期结束`
- `raw/` 存放员工原始周报文件和未通过校验的文件
- `apv/` 存放校验通过的周报文件
- `dept/` 预留给 `merge-reports` 存放生成好的部门周报

每次执行应面向 1 个明确的周目录；不同周期的周报应事先分别放入各自的 `monthYYYYMM/weekYYYYMMDD/raw/` 后再校验。

## 执行步骤

### Step 1：收集待校验文件

- 如果输入是标准周目录 `monthYYYYMM/weekYYYYMMDD/`，只扫描其下 `raw/` 中待校验的 `.md` 文件。
- 如果输入是标准 `raw/` 目录，只扫描该目录下待校验的 `.md` 文件，并将其父目录视为当前周目录。
- 当前标准流程不再以单个散落文件作为主要输入；如用户只给单个文件，应先提示其放入对应周目录的 `raw/` 后再执行。
- 进入校验前，必须先按文件名模式 `人名工作周报YYYYMMDD.md` 过滤；不匹配该模式的文件直接跳过，不纳入校验范围，但仍保留在 `raw/`。
- 如果周目录缺少 `raw/`、`apv/` 或 `dept/`，直接提示目录结构不完整。
- 如果过滤后没有待校验文件，直接输出“未找到符合命名模式的周报文件”。

### Step 2：读取规范

读取 `references/个人周报格式.md`，确认以下规则后再开始校验：
- 文件名格式
- YAML front matter 字段和日期格式
- 填报周期规则（固定为上周五至本周四）
- 模块顺序、已填写板块顺序和标题层级
- `本周工作` 与 `下周工作` 的不同表头要求
- 空板块写法
- 日期格式要求
- 当前状态建议值

### Step 3：逐文件校验

对每份周报，依次检查以下各项：

#### 2.1 文件名
- 是否符合 `人名工作周报YYYYMMDD.md` 格式
- YYYYMMDD 是否为有效日期
- YYYYMMDD 是否与 front matter 中 `填报周期结束` 一致

#### 2.2 YAML front matter
- 是否存在且位于文件开头
- 是否包含 `姓名`、`填报周期开始`、`填报周期结束` 三个字段
- 日期是否为 `YYYY-MM-DD` 格式
- `填报周期结束` 是否晚于或等于 `填报周期开始`
- `填报周期开始` 是否为周五，`填报周期结束` 是否为周四
- `填报周期开始` 与 `填报周期结束` 是否正好相差 6 天，构成完整周报周期

#### 2.3 模块、板块完整性与顺序
- 是否包含全部 2 个模块：`本周工作`、`下周工作`
- 模块顺序是否正确
- 已出现的业务板块名称是否属于固定集合：金融研发、寄递研发、管理赋能、数据分析、信息资源管理、队伍建设及业绩融合
- 已出现的业务板块顺序是否正确
- 同一模块内是否重复出现同一个业务板块
- 模块标题是否为二级标题 `##`
- 已出现的业务板块标题是否为三级标题 `###`
- 未出现的业务板块默认视为 `无`，不计为问题

#### 2.4 空板块写法
- 无内容的业务板块是否在标题下写了单独一行 `无`
- 无内容的业务板块是否错误地写了空表格或直接空白
- 未出现的业务板块是否按默认 `无` 处理

#### 2.5 表格格式
- 有内容的业务板块是否使用了标准 Markdown 表格
- `本周工作` 模块下各板块表头是否完整：`项目名称 | 开始时间 | 结束时间 | 当前状态 | 工作内容 | 工作成果`
- `下周工作` 模块下各板块表头是否正确：`项目名称 | 开始时间 | 结束时间 | 当前状态 | 工作内容`
- 分隔行是否为标准格式 `| --- | --- | ... |`

#### 2.6 日期字段
- 每行 `开始时间` 是否为 `YYYY-MM-DD` 格式
- 每行 `结束时间` 是否为 `YYYY-MM-DD` 格式或空（进行中可留空）
- `下周工作` 模块下每行的 `开始时间` 是否晚于 `填报周期结束`

#### 2.7 当前状态
- 建议值范围为：`已完成`、`进行中`、`已提交待确认`、`已上线`、`已提测`、`待启动`
- 如果使用了非建议值，输出警告，不计入错误数量

#### 2.8 内容规范
- `工作内容` 是否含有模糊表述，如：`做了一些优化`、`处理相关问题`、`持续推进若干工作`
- `本周工作` 模块中的 `工作成果` 与 `工作内容` 是否明显重复
- 一行中是否混入多个项目名称或多个部门名称

### Step 4：输出问题清单

每份文件的校验结果按以下格式输出：

```
## 文件：郭思文工作周报20260326.md

通过项目：文件名、front matter、模块结构、表格格式、日期格式

问题列表：
- [2.6 日期字段] 下周工作 > 管理赋能 > 文创产品进销存系统：开始时间 `2026-03-24` 未晚于填报周期结束日期 `2026-03-26`

警告列表：
- [2.7 当前状态] 本周工作 > 寄递研发 > 车牌寄递辅助系统：当前状态 `在途` 不在建议值范围内，建议改为标准表达

结论：存在 1 个问题，需要修改后重新提交。
```

如果文件全部通过，输出：

```
## 文件：郭思文工作周报20260326.md

全部通过，格式合规。

警告列表：
- 无
```

### Step 5：整理合规文件

对 `raw/` 中的校验结果执行目录内整理：

- 当前步骤不再额外复制文件，也不重新分发到新的周目录。
- 校验结果为“无问题”的文件，从当前周目录的 `raw/` 移动到同级 `apv/`
- 存在警告但无问题的文件，也从 `raw/` 移动到 `apv/`
- 存在问题的文件，继续保留在当前周目录的 `raw/`
- 文件名不符合模式而被跳过的文件，继续保留在 `raw/`，不进入 `apv/`
- 如果 `apv/` 中已存在同名文件，应以本次通过校验的文件覆盖旧版本

输出时需明确列出：

- 当前处理的是哪个 `monthYYYYMM/weekYYYYMMDD/` 目录
- 哪些文件从 `raw/` 移动到了 `apv/`
- 哪些文件因存在问题继续保留在 `raw/`
- 哪些文件因文件名不符合模式被跳过并保留在 `raw/`

### Step 6：汇总

所有文件校验完成后，输出一行汇总：

```
校验完成：共 N 份文件，M 份合规，K 份存在问题，W 份存在警告。
```

## 注意事项

- 校验时以规范文件为准，不要凭记忆判断规则。
- 日期合法性校验需考虑月份天数（如 2 月不超过 28/29 天）。
- 填报周期校验按固定规则执行：`填报周期开始` 为周五，`填报周期结束` 为下周四。
- 模糊表述检测不要过度误报，仅标记明显的无实质内容描述。
- 问题描述需定位到具体模块、板块和行，方便填报人修改。
- 未出现的业务板块统一视为 `无`，不要求填报人手工补齐所有空板块标题。
- 排序规则留待后续“周报合并” skill 处理，本 skill 不检查排序。
- 当前推荐流程是：员工先放入 `raw/`，校验通过后移动到 `apv/`。
- 目录输入场景下，只有文件名匹配 `人名工作周报YYYYMMDD.md` 的文件才纳入校验统计。
- `merge-reports` 默认只读取 `validate-report` 输出的 `monthYYYYMM/weekYYYYMMDD/` 目录，并从其中的 `apv/` 读取合规周报，不再重复承担个人周报格式校验职责。
